这套方案的优点
迭代拆分合理：9（连接）→10（推送）→11（主动触发），节奏清晰。
Actor 统一抽象一致：User/Agent 共用通信模型，后续权限模型更统一。

建议（可执行）
安全优先
WebSocket 认证改为 Authorization 头或 Sec-WebSocket-Protocol，不要走 query string。
明确关闭码规范：认证失败用 1008，内部错误 1011。
可靠性增强
引入“会话游标/最后已收消息ID”模型；ACK 落库（至少 per actor per conversation）。
断线重连提供 since_message_id 补拉协议。
多设备在线语义修正
仅当“最后一个连接断开”才广播 offline；首个连接建立才广播 online。
并发与幂等
Trigger 执行加 SELECT ... FOR UPDATE SKIP LOCKED 或分布式锁。
SendMessage 加幂等键（client_msg_id），避免重试重复写入。
扩展路线明确化
在文档里补一条“生产切换条件”：当多实例部署时，EventBus/连接状态切到 Redis（Pub/Sub + presence）。
观测与限流
增加指标：连接数、推送成功率、P95 推送延迟、ACK 延迟、trigger 成功率。
增加 WS 消息大小限制、每连接频率限制、防刷策略。
